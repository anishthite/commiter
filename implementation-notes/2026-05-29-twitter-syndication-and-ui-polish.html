2026-05-29 — Twitter syndication source + dashboard UI polish

Scope. User reported four things at once: (1) tab title was the Evangelion in-joke (NERV // Central Dogma) and should be replaced; (2) day-streak number was zero-padded to 3 digits (004) — drop the pad; (3) on desktop the panel content was cramped against a narrow container with the streak/today/longest stats stacked in a tall column, even though there was 800+px of unused horizontal space on each side; (4) Twitter feed was returning offline on prod with all 4 Nitter hosts failing.

Decisions

IDWhatWhy
D-018Switch Twitter primary source from Nitter pool to syndication.twitter.com/srv/timeline-profile/screen-name/<handle>; keep Nitter as fallback.Live probe showed all 4 Nitter hosts failing from Vercel egress IPs (DNS fail on nitter.net, Cloudflare challenge on poast/privacyredirect, RSS allowlist on xcancel). Same probe found syndication endpoint returning 200 + 427 KB HTML with __NEXT_DATA__ embedding 101 recent tweets, no cookies. Contradicts the 2026-05 scout doc which claimed it was cookie-gated — endpoint behavior has evidently shifted.
D-019Use node:https (not fetch) for the syndication call.Direct A/B from same IP, same second: curl = 200, node:https.get = 200, but Node's built-in fetch (undici) = 429 (“Rate limit exceeded”, body length 20) on the very first request. Twitter is fingerprinting undici's TLS handshake. Node core OpenSSL bindings present a different JA3/JA4 that gets through. Vercel runs the same Node binary, so the bypass carries over.
D-020Bypass Next.js fetch-level cache for the syndication call. Rely on page/route-level revalidate = 3600 instead.node:https isn't wrapped by Next's caching layer (that's fetch-only). Acceptable because the rendered snapshot is itself cached for 1h, so we still only hit the endpoint once per region per hour — well under any plausible quota.
D-021Widen page container max-w-3xl (768px) → max-w-6xl (1152px); within each MagiPanel, lay out the three stats horizontally (flex sm:flex-row) on desktop with the heatmap below.User explicitly: “we have a lot more space on left and right we can use.” Stacking stats vertically was a holdover from mobile-first; on desktop with the wider container there's plenty of room for a row layout. Heatmap stays narrow + tall (week-major) — that's a separate redesign if we want it horizontal.
D-022Drop padStart(3, "0") on streak; render the raw integer.User feedback: “004” reads as a serial number, not a streak count. Removed pad makes the number self-evident; tabular-nums keeps alignment between panels regardless of digit count.
D-023Tab title: NERV // Central Dogma → did we ship today?Matches the H1 on the page; what users actually read in browser tabs is the title, and the H1's question framing is the point of the whole app.

Tradeoffs

IDTradeoff
T-014Syndication endpoint returns ~101 tweets max (vs Nitter's 20). Strict upgrade for count fidelity, but still undercounts heavy posters for the 365-day heatmap. No paginated unauth variant exists. Mitigated by: most relevant signal is recent days + today, both well within 101 entries even for 5+ tweets/day cadence.
T-015TLS-fingerprint workaround (node:https instead of fetch) is durable for now but fragile in principle — if Twitter starts fingerprinting at a deeper layer (e.g., browser-specific JA3 only), this collapses. Followup L-015 tracks the fallback ladder.
T-016Wider container breaks the “mobile-first compact card” aesthetic on tablet sizes (640–1024px) — stats jump from column to row at the sm: breakpoint (640px) while the heatmap stays the same width below. Acceptable: panels still read top-to-bottom as a coherent unit, just sparser.

Late pivot (same day, post-deploy verification)

Deployed; user reported the prod heatmap was missing the last ~4 months of tweets even though they had posted. Diagnosed by probing four daily-active control accounts via the same endpoint:

HandleSyndication "newest" date
@paulg2025-10-30 (7 months stale)
@sama2025-11-14 (6 months stale)
@dhh2025-10-20 (7 months stale)
@levelsio2026-05-25 (1 in last 30d, this handle tweets ~20×/day)

Conclusion: syndication.twitter.com/srv/timeline-profile serves a curated/cached embed-widget timeline, not a chronological live feed. It's not a rate-limit issue, not a fingerprint issue, not a query-param issue — it's the endpoint's literal job to surface ~100 "popular tweets" for widget embedding. Wrong tool for the "did I ship today?" use case.

Replacement decisions

IDWhatWhy
D-024Promote rettiwt-api v7 to PRIMARY Twitter source, demote syndication to fallback, keep Nitter as last-resort.rettiwt-api hits X's real /UserTweets GraphQL via authenticated cookies. Chronological, real-time. Guest mode confirmed dead 2026-05-29 (DataValidationError on user.timeline, "Not authorized" on tweet.search) — a user-supplied API_KEY (base64'd auth cookies from X Auth Helper browser extension, stored in X_RETTIWT_KEY) is mandatory.
D-025Keep syndication as fallback even though it's stale.When X_RETTIWT_KEY is unset (e.g., fresh fork, key not yet provisioned), "a few popular tweets from months ago" beats "all zeros forever." The user-visible failure mode is "recent days look quiet" instead of "this app is broken."
D-026Paginate up to 5 pages OR until oldest item in a page is older than the from cutoff, whichever comes first.Bounds worst-case latency for heavy tweeters (~1000 tweets max scanned). For normal cadence this comfortably covers the 365-day heatmap window.

Tradeoffs (new)

IDTradeoff
T-017API_KEY = base64'd live cookies. Treat it like a password. Vercel env vars are encrypted at rest, which is the right place. Documented this caveat prominently in .env.example so future-me doesn't shove it in a public repo.
T-018rettiwt-api is reverse-engineered against X internals. Twitter changes endpoint hashes every few months; the package author chases them. Expect ~quarterly version bumps. Acceptable for a hobby tracker; would not ship this for SLA-critical use.
T-019API_KEY expires when the user explicitly logs out of the X session that generated it. Refresh path is "redo the 2-minute extension flow." Documented in .env.example.

Second pivot (same day, post-rettiwt push)

User pushed back on rettiwt-api: “this one looks sus.” Threat-model objection is correct — putting live X auth cookies in a Vercel env var means anyone with access to the project's env (you, an exfiltrated build log, a Vercel team-member compromise) gets full account takeover of your real X account. Reverted same-day.

D-027 — final source ladder

Removed rettiwt-api dependency (pnpm remove), stripped fetchRettiwtDays from twitter.ts, swapped X_RETTIWT_KEY for X_NITTER_HOST as the user-supplied primary source. Final ladder:

TierSourceWhere the X cookies live
1Self-hosted Nitter at X_NITTER_HOSTUser's Fly.io VM, burner X account (not real)
2syndication.twitter.com (stale curated, kept as soft fallback)n/a
3Public Nitter RSS pool (residential-IP-only)n/a
4throw → zero-fill snapshotn/a

Wrote docs/SELF_HOST_NITTER.md as the 1h Fly.io deployment walkthrough. Key threat-model improvements over rettiwt-api: (a) cookies live on a VM the user controls, not in Vercel's env, (b) cookies are from a burner X account with no real-life ties, (c) revocation = log the burner out, no Vercel involvement.

T-020 / T-021 / T-022

IDTradeoff
T-020Burner X account must NOT have 2FA (Nitter's session importer can't handle 2FA flows). Acceptable because the burner has nothing of value; documented this caveat in the deploy guide.
T-021Self-hosted Nitter is ~1h of one-time setup work for the user. Significantly more than "paste a key into Vercel." Justified by the elimination of real-account-takeover risk and zero ongoing $ cost.
T-022Cold-start latency on Fly's free tier when the Nitter machine is auto-stopped: 1–3s on the first request after idle. Acceptable because our snapshot path is hit hourly (revalidate=3600), so most user page loads hit the warm cache; only the cron-style revalidation pays the cold start, and it's still well under our 8s fetch timeout.

Third pivot — ship by getting out of the scraping business entirely

After the rettiwt rollback, user surfaced the bigger picture: they want to do other projects (in-Chrome LLM-driven Twitter automation, style-mimicry tweet model, follower analysis, etc.) and correctly identified that any server-side scraping path is doomed because anti-bot in 2026 has won. The right move isn't to make commiter smarter at scraping — it's to make it dumber. Decouple the scraping concern from the rendering concern permanently.

Removed syndication.twitter.com entirely (D-028) — silently-wrong stale data is worse than honestly-empty. With nothing wired up, the dashboard now renders a clean GitHub-only view with a wide GitHub MagiPanel and a one-line subscript explaining how to enable Twitter. The combined streak switches from AND to OR-by-degeneration when Twitter is offline (“did we ship today?” answers from GitHub alone, doesn't punish the user for not configuring a Twitter source).

T-023 / T-024

IDTradeoff
T-023Wire format is the integration surface and we own zero of the producers. Chose a forgiving shape (accept bare array OR {days: array}, ignore unknown fields, drop out-of-window dates silently) so producers can ship richer payloads (timestamps, engagement counts, hashtags) without breaking commiter. The cost: looser validation means a malformed producer could silently produce zero-fills. Acceptable because the producer is the user's own code.
