2026-06-01 — Multi-user dashboard: anish + subby, with mobile-optimized summary

Scope. Today the app tracks one user (anish) at /. User wants: (1) track a second user, "subby" (GitHub thecooltechguy, X subby_tech); (2) the existing single-user view becomes /[user]; (3) / becomes a high-level summary showing both users side-by-side with a small per-user heatmap; (4) the whole thing mobile-optimized.

User-confirmed product decisions (ask_user_question, 2026-06-01).

  • Config shape: apps/web/src/config/users.json committed to the repo.
  • Subby's handles: GH thecooltechguy, X subby_tech.
  • Summary cell content: name + shipped today (yes/no) + combined streak + tiny heatmap.
  • Mini heatmap: last 90 days, ~8px cells, mobile-first horizontal strip.

Decisions

IDWhatWhy
D-024User roster lives in apps/web/src/config/users.json, with shape { users: [{ slug, displayName, githubLogin, xLogin }] }. Imported directly by both the runtime composer and the GH Action script. Single source of truth.User explicitly chose this option over hardcoded array or env-var fanout. Keeps additions to a one-file edit and survives process.env being unset.
D-025Per-user X data files: apps/web/src/data/x-days.{slug}.json. Migration: rename existing x-days.json → x-days.anish.json (it already holds anish's data); create empty x-days.subby.json seed so the panel renders on first deploy.Same file shape as today, just keyed by slug. Migration is a rename, not a schema change — preserves all of anish's backfilled history. The empty seed for subby lets the renderer treat his X panel as "offline until refresh runs" (same code path the original repo already handles).
D-026getSnapshot takes an explicit { slug, githubLogin, xLogin } argument instead of reading env. Env stays only as a fallback for the legacy single-user path.The old code hardcoded process.env.GITHUB_LOGIN/X_LOGIN; that no longer matches a multi-user model. Passing the resolved user explicitly makes the composer a pure function of its input and is testable.
D-027Routes: / = summary page (both users); /[user] = per-user detail (the current page).Matches the user's literal request. URL slugs are user.slug (e.g. /anish, /subby). generateStaticParams can pre-render both routes since the user list is static at build time.
D-028Summary cell content: name, big "yes."/"no." for shipped-today, combined-streak count, tiny 90-day heatmap. Click/tap whole cell → drill into /[user].User-confirmed option. Compact enough to fit two cells on a phone (stacked on narrow viewports, side-by-side on tablet+).
D-029New MiniHeatmap component, separate from existing Heatmap: last 90 days, ~8px cells with 1px gap, no axis labels, no tooltips. Reverse-chrono horizontal strip (newest right, oldest left).Reusing Heatmap directly would inherit its month/day labels, tooltips, and 14px cells — too heavy for a summary cell. A separate stripped-down component is simpler than parameterizing the existing one with 6 toggle props.
D-030Refresh script (scripts/refresh-x-days.ts) and workflow updated to loop over users.json: read/write each user's own x-days.{slug}.json. One workflow run = one commit, refreshing every user's file in a single pass.Splitting per-user runs would multiply commits, increase API quota burn, and complicate the deploy-hook trigger. A single sequential loop is fine for n=2; if we ever scale past ~10 users we'll revisit.
D-031Mobile layout: summary cells stack vertically below sm breakpoint (640px), side-by-side at sm+. Each cell is a full-width tap target on mobile with the heatmap below the stats. Within /[user], the existing layout is already mobile-friendly — preserve it.Two cells side-by-side at 320px viewport width = ~150px per cell, not enough room for a 90-day heatmap strip (90×9px = 810px). Stacking is the only honest fit on phones.

Tradeoffs

IDWhatCost
T-014Summary / page fetches snapshots for BOTH users in parallel, every render.2× the GitHub GraphQL calls per render. Mitigated by the existing 1h fetch cache + 1h page revalidate; ceiling is ~24 GH calls/user/day from cold deploys, which is well below the 5000/hour rate limit on a single PAT.
T-015Mini heatmap doesn't show today's date or tooltips.Less information on the summary. Intentional — the drill-in route has the full heatmap with tooltips; the summary's job is "did they ship?" at a glance, not data inspection.

Reversibility

IDNote
R-008If multi-user proves unwanted, revert by: (a) restore page.tsx from this commit's parent, (b) rename x-days.anish.json back to x-days.json, (c) delete /[user] route, (d) delete users.json and MiniHeatmap.tsx. The data files are forward-compatible: anish's full backfill survives the rename round-trip.

Followups

IDWhat
L-007The legacy env-fallback in resolveUser() synthesizes a user with slug: "env" when GITHUB_LOGIN doesn't match a roster entry. X_DAYS_BY_SLUG only registers roster slugs, so a fork using only env vars would silently hit TwitterFeedOfflineError and hide the X panel. Acceptable for our use (we always go through the roster) but worth flagging in README before the next forker hits it.
L-008Reviewer noted the workflow commit message is now generic ("refresh x-days") even when only one user's data changed. Easy enhancement: capture script stdout, parse the "ok slug=..." lines, build a richer commit message. Not blocking.

Open questions

IDQuestion
Q-006Should the summary headline read "did we ship?" (collective) or list each user separately? Going with the latter (two cells, no collective question) since the existing per-user page already handles the question framing.
Q-007Are both users using the same GITHUB_TOKEN PAT? Assumed yes — a single PAT with read:user can pull any user's public contributions calendar via GraphQL. Private contribs only show through for the token's owner. Acceptable since subby is being tracked, not authenticated as.
