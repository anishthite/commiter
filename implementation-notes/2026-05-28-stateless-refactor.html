Stateless refactor — drop Turso, drop pi-chrome, drop ingestor

main · worktree ~/workspace/commiter · 2026-05-28 · status implemented · blast medium

Summary

Goal
Replace the two-machine architecture (Mac ingestor → Turso → Vercel reader) with a single Next.js app that fetches GitHub GraphQL + X syndication endpoint directly on each SSR, cached 1hr.
User intent
"simplify… nice free twitter api and github api, cache it/invalidate once an hour" — and explicit answer "Fully stateless, no DB at all" + "Forward-only is fine" for Twitter history.
Supersedes
2026-05-23 plan §2 (Twitter via pi-chrome), §3 (Architecture), §4 (Data model), §6 phases 2 & 4, §7 (Repo layout).
Survives unchanged
GitHub source = GraphQL contributionsCollection. Streak math. TZ = America/Los_Angeles. AND-combined streak (but see D-003). NERV aesthetic + components.
Deferred
Nudge mechanism + ship CLI manual fallback (user said "don't worry about for now").

First end-to-end run revealed three layered bugs that the worker's sandbox could not have caught. All fixed inline:

X-002Host order had two dead hosts ahead of one working host.I propagated stale researcher recommendations; live probe of all 6 known Nitter instances gave the correct ranking.Reordered to [nitter.net, nitter.poast.org, xcancel.com, nitter.privacyredirect.com]. nitter.net is the only currently-reliable primary as of 2026-05-29.

Worker report (2026-05-28)

Additional decisions made during implementation

IDDecisionWhy
D-011D-008 implementation: the leading buffer is implemented in github.ts by extending the GraphQL ISO window [from, to] by ±1 day from the PT-aligned range, then trimming via fillMissingDays(rows, from, to). snapshot.ts still computes from = addDays(today, -(days - 1)) in PT and passes that as the displayed range.Confines the buffer hack to the fetcher; snapshot.ts stays clean.
D-012fetchTwitterDaysSafe (private helper inside snapshot.ts) also catches the case where X_LOGIN is empty — logs a warning and returns zero-filled days. Not a hard error.Lets a freshly cloned repo render the GitHub panel with only GITHUB_TOKEN + GITHUB_LOGIN set. Reduces onboarding friction. The error path for missing token is hard (GitHub throws an inline friendly error) because the entire dashboard relies on it.
D-013RSS sanity check: a host's response only counts if the body matches /<item[\s>]/ or /<entry[\s>]/. Cloudflare bot-challenge HTML pages fail this check and we fall through to the next host. Empirically validated against nitter.privacyredirect.com (returns 200 + HTML challenge) and nitter.poast.org (returns 403 + browser challenge).HTTP 200 is not sufficient evidence of valid data here. Content-type alone isn't either — some Nitter hosts omit it.
D-014Followed spec literally on the page.tsx revalidate vs force-dynamic question: removed force-dynamic, added revalidate = 3600. Flagged the build-time prerender wart as L-004 rather than silently deviating to force-dynamic.Spec was explicit (S15). Operational nuance is real but escalable; not the worker's call.

Followups discovered

  • L-004 — (added above) build-time prerender wart: SYS:FAULT may be cached for ~1h post-deploy if GITHUB_TOKEN isn't present at build. Workaround options: set env at build, or flip page back to force-dynamic. Documented in PLAN.md §7.
  • L-005 — fetchGithubDays doesn't honor requireGithubToken's old behavior of dumping a multi-line friendly message including PAT scope hints. Current message is one line. Fine for now but worth restoring the longer guidance when error UX gets a polish pass.
  • L-006 — Nitter RSS gives only the most recent ~20 tweets. Once you've shipped multiple days of posts the X heatmap will look usefully sparse; users with bursty posting cadence will see big grey gaps that are technically honest but visually misleading. Consider either (a) graying out cells beyond the feed window with a distinct color, or (b) adding the L-003 Upstash rollup.
