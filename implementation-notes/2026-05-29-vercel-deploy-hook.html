2026-05-29 — Vercel deploy hook for refresh-x-days bot

Scope. Vercel's git-push auto-deploy was refusing builds triggered by the refresh-x-days GitHub Action. The action's daily commit was being attributed to GitHub's catch-all web-flow user (because user.email was the unspecific noreply@github.com), and that identity has no Vercel account linked. Vercel's project setting "deploy only commits from authors with access to this project" then blocked the build. Result: apps/web/src/data/x-days.json updated on main daily but the deployed site never picked up the new data.

Decisions

IDWhatWhy
D-001Trigger Vercel via Deploy Hook instead of fixing the commit-author attribution.User picked option 3 from a 4-option clarify. Author-attribution fixes (use real noreply email, or onboard github-actions[bot] as a project member) are smaller diffs but tie the deploy path to identity plumbing that drifts. A deploy hook is one POST; its auth is the URL itself; it bypasses the author check by design. The hook URL is project- and branch-scoped, so even if leaked it can only trigger a redeploy of the current main, never inject code.
D-002Gate the curl on steps.commit.outputs.committed == 'true' so no-op runs don't burn a Vercel build.Most days the latest tweet already exists in the JSON and git diff --quiet exits 0 — no commit, no push, nothing for Vercel to deploy. Free-tier Vercel build minutes are finite and a daily no-op deploy would consume them for nothing.
D-003Treat a missing VERCEL_DEPLOY_HOOK_URL secret as a soft skip (warn + exit 0), not a hard fail.Workflow must keep working on forks and on first-deploy where the secret hasn't been provisioned yet. The commit still happens; only the deploy trigger is skipped. Loud log line tells the operator what to do.
D-004Use curl -fsS (fail on non-2xx) for the POST itself.If the hook is rotated or deleted, Vercel returns 404. We want that to fail the job so the operator notices on the next dashboard glance, not silently swallow the failure.
D-005Leave user.email = noreply@github.com as-is.With deploy hooks, the commit author no longer gates the deploy. Changing the email would also re-attribute the historical commit log; not worth the churn when the new path already works.
D-005a2026-05-31 revision: reversed D-005. Switched user.email to 14373068+anishthite@users.noreply.github.com (the repo owner's GitHub noreply, numeric-id-prefixed form).D-005 misread the failure mode. Vercel's git integration still attempts an auto-deploy on EVERY push to the production branch, in parallel with the deploy hook — the hook does not suppress it. For bot commits authored as noreply@github.com, the parallel auto-deploy is refused with "Git author web-flow must have access to the project on Vercel," generating a recurring failure notification per refresh. The hook deploy succeeded silently underneath, so the site updated correctly, but the user kept getting the noise. The numeric-id-prefixed noreply maps to the actual GitHub user (which has Vercel access), so the auto-deploy is accepted and no error fires. The hook step stays as belt-and-braces.

Tradeoffs

IDTradeoff
T-001Two deploy paths now exist for main: Vercel's git auto-deploy (which is refused for bot commits but still fires for the operator's own commits) and the hook. For human commits, the auto-deploy wins and the hook never runs (bot only). For bot commits, the auto-deploy attempt is refused (no-op) and the hook drives the build. No double-deploy; no race. If Vercel's auto-deploy is later disabled at the project level for cleanliness, human commits will need their own trigger — flagged as L-001.
T-002Hook URL is a long-lived bearer-style secret. Leaking it = attacker can spam redeploys of current main. Cost: build minutes. Mitigation: standard GitHub Actions secret handling; if it leaks, regenerate in Vercel UI and rotate the secret. No data exfil path through this primitive.

Open questions / followups

IDItem
L-001Optional: disable Vercel's git-push auto-deploy entirely (Project → Settings → Git → Ignored Build Step or Production Branch toggle), and trigger all deploys via hooks. Cleaner separation, but every human commit then needs an action/manual deploy. Not doing this now — current hybrid works.
L-002If Vercel ever exposes a commit-sha parameter on deploy hooks, pass the freshly-pushed SHA to guarantee the hook builds the bot's commit and not a later one that landed mid-flight. Today the hook just builds tip-of-branch; the rebase-before-push step (F7) keeps this race window small.
L-003With D-005a in place, the F8 deploy-hook step is now redundant for the happy path — Vercel's git auto-deploy handles bot commits correctly. Kept as defense-in-depth: if the Vercel git integration is ever disconnected (or the project is moved to a Vercel team without git linkage), the hook still drives deploys. Worth re-evaluating after a month of clean runs: drop F8 and the VERCEL_DEPLOY_HOOK_URL secret if the auto-deploy path proves stable.

Rollout checklist

  1. In Vercel: Project → Settings → Git → Deploy Hooks → Create Hook. Name: refresh-x-days-bot. Branch: main. Copy URL.
  2. In GitHub: repo → Settings → Secrets and variables → Actions → New repository secret. Name: VERCEL_DEPLOY_HOOK_URL. Value: paste URL.
  3. Run refresh-x-days via workflow_dispatch. If the JSON has no diff that minute, force a diff first (e.g., bump the file by hand and revert in a single commit) — or just wait for the next scheduled run.
  4. Confirm in Vercel dashboard: a new deployment kicked off, labeled "Deploy Hook" as the trigger source.
  5. Verify https://<site>/ reflects the updated x-days.json.
