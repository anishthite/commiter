2026-05-29 — X data source migration: live scrape → bundled JSON via daily GH Action

Scope. Replace the runtime live-scrape ladder in apps/web/src/lib/twitter.ts (public RSS pool, self-hosted RSS, X_DATA_URL HTTP fetch) with a single bundled apps/web/src/data/x-days.json imported at build time. A new GitHub Action (.github/workflows/refresh-x-days.yml) runs daily, paginates the user's recent tweets from api.socialdata.tools, buckets each into a NERV_TZ day, merges into the JSON, and commits back to main — Vercel rebuild ships the new data. The runtime app no longer touches any X API.

Decisions

IDWhatWhy
D-029Delete tiers 2 (self-hosted RSS via X_NITTER_HOST) and 3 (public RSS pool: nitter.net / nitter.poast.org / xcancel.com / nitter.privacyredirect.com) entirely.Both were already known-broken from Vercel egress (Cloudflare challenges, DNS fail, RSS allowlist sentinels). Keeping them as "defense in depth" was net-negative: dead code that ran on every cache miss and produced misleading attempts arrays in offline logs. The new architecture has zero live network calls at render time, so the fallback ladder collapses to "is the bundled JSON valid?".
D-030Use a bundled JSON imported via import xDaysData from "../data/x-days.json" rather than (a) fs.readFile at runtime or (b) fetch(raw.githubusercontent.com).The repo's tsconfig.base.json already enables resolveJsonModule, so the import is a one-liner with full TS narrowing. Bundling the JSON into the Next.js build output means render-time has zero I/O and zero network — the data IS the deploy. fs would work but adds an async boundary for nothing; fetch reintroduces the cache-coherence problem the migration is trying to escape (the whole point is that the deploy and the data are atomically versioned).
D-031Treat empty days[] as offline (throw TwitterFeedOfflineError) rather than rendering an all-zero heatmap.Empty days is the legitimate first-deploy state before the GH Action has ever run. snapshot.ts's existing catch-and-degrade path hides the panel and switches combined mode to "or" — exactly the right behavior. Rendering all-zeros would silently lie ("I posted nothing in 365 days") which is the same antipattern the syndication.twitter.com tier (removed in D-028) created.
D-032Preserve X_LOGIN, FetchTwitterOpts, fetchTwitterDays, TwitterFeedOfflineError as the public surface — signature-identical to the pre-migration module.The one consumer (snapshot.ts:132) doesn't need to change. X_LOGIN is still used for panel-presence logic (empty → panel hidden, snapshot.ts:126). The tz param is now unused at read time (the refresh script does the bucketing) but kept so the call site is byte-identical.
D-033Refresh script lives at scripts/refresh-x-days.ts, invoked via pnpm tsx scripts/refresh-x-days.ts. tsx added at the workspace root (not as a per-package dep).The script is workspace-level glue, not an apps/web concern. Putting tsx at the root keeps apps/web/package.json focused on the deployed app. tsx over ts-node because tsx is ESM-native and faster cold-start — matters for a GH Action where install + run is the entire wall-clock budget.
D-034Incremental refresh: since = max(existing date) − 2 days; full backfill from BACKFILL_SINCE (default 2024-01-01) only when days[] is empty. Within the 2-day overlap, new fetch REPLACES existing counts. Outside the overlap, days are NEVER decremented.Two-day overlap absorbs late-arriving tweets (deletes, edits, socialdata indexing lag) without requiring a full re-pull. Never-decrement-outside-overlap protects against socialdata's quota-trimmed responses silently zeroing out historical data — the bundled JSON is the cumulative source of truth, not the API.
D-035Refresh script's dateKey() uses Intl.DateTimeFormat("en-CA", { timeZone: NERV_TZ, year: "numeric", month: "2-digit", day: "2-digit" }) — byte-identical to streak.ts:dateKey().The TZ boundary is the single most fragile invariant in the system. If the producer and consumer disagree on the bucket-day for a given UTC timestamp, the heatmap silently misaligns. Copy-pasting the exact Intl call into the script (rather than importing it) is intentional: the script must keep working as a standalone tsx run with no @/lib path resolution.
D-036Cron at 0 9 * * * (09:00 UTC ≈ 01:00 PST / 02:00 PDT). Default GITHUB_TOKEN with contents: write permission for the push; no PAT.09:00 UTC is well past midnight PT, so the prior day's tweets are settled by then. GITHUB_TOKEN avoids the secret-rotation overhead of a PAT and works for any repo without branch-protection rules requiring a verified author — if branch protection is added later, this would need to be revisited.

Tradeoffs

IDTradeoff
T-017Data freshness is now bounded by the daily Action + Vercel deploy cycle. Tweets posted at 18:00 PT show up the morning after the next 01:00–02:00 PT refresh, then take another minute or two to ship through Vercel. Worst case: ~26-hour latency. Acceptable for a "did I ship today?" tracker; if the user wants sub-hour freshness they can workflow_dispatch manually after posting.
T-018SOCIALDATA_API_KEY is a paid-API dependency. If the key is revoked or socialdata.tools goes down, the daily refresh starts no-op'ing (script exits 1, no commit, JSON ages). The dashboard keeps rendering stale data indefinitely — there is no "data is stale" warning. Mitigation deferred: a future enhancement could add a generated_at-age check in twitter.ts and degrade after N days.
T-019JSON file lives inside the Next.js build output, so even a one-character change to the data file triggers a full Vercel rebuild. At today's repo size (small) this is ~30s of CI; if the repo grows large this becomes friction. Could be revisited by moving the JSON to a separate sibling repo and fetching via raw.githubusercontent.com with next: { revalidate } — but that re-introduces the cache-coherence problem D-030 deleted.
T-020Script's since: arg to socialdata's search is interpreted on UTC dates, but we re-bucket each tweet to NERV_TZ regardless. There's a ~17-hour edge band where a tweet's UTC date differs from its PT date, but the 2-day overlap window absorbs it. If OVERLAP_DAYS were tightened to 0, this would silently lose tweets near midnight PT.

Open Questions

IDQuestion
Q-006Will GITHUB_TOKEN with contents: write work for pushing back to main on this repo? If branch protection rules require a verified signature or status checks, the workflow will fail at push time and we'll need a PAT (or a deploy key, or a GitHub App). Not testable from the worker session — first scheduled run will surface it.
Q-007What's socialdata.tools' rate limit / cost-per-call? Script paginates up to 200 pages on full backfill; if each page is metered, the first run could be expensive. Could be worth capping MAX_PAGES tighter or staging the backfill across multiple runs.
Q-008Does socialdata's since: filter include or exclude the boundary day? Treating it as inclusive in the merge logic; if it's exclusive there's a 1-day gap that the 2-day overlap absorbs anyway. Worth verifying after the first real run.

Review Fixes (post-merge)

Three adversarial reviewers (correctness, GHA security, robustness) ran against the worker's diff. One BLOCKER + three HIGHs + six MEDs landed; the fixes below were applied as F1–F10.

IDSevReviewerFix
D-037 (F1)HIGHcorrectnessmergeCounts's fresh-copy loop now guards merged.set(date, count) with if (fullBackfill || date >= since). Pre-since PT dates that arrived as partial counts (UTC ⇔ PT boundary noise) no longer clobber the prior full-day totals. The day before the overlap window is now genuinely never touched by an incremental merge.
D-038 (F2)BLOCKERrobustnessThree-layer defense against silent data loss when the API returns 200 + non-search-shaped body. (1) apiGet now throws on {status:"error"} / {error:…} envelopes after JSON parse. (2) fetchTweetDayCounts throws on the first page if body.tweets is undefined (not just empty). (3) main() refuses to write when !fullBackfill && mergedDays.length < existing.days.length — any genuine shrink requires an explicit re-backfill. False-positive risk on legit "user posted nothing" runs is acceptable; the brief flagged this and chose loud-failure over silent-shrink.
D-039 (F3)HIGHrobustnessapiGet now retries on 429 + 5xx with exponential backoff (1s, 4s, 16s) up to 3 retries (4 attempts total). 429 honors Retry-After seconds if present. 4xx other than 429 throws immediately (auth/bad-request — not transient). Network-level fetch throws are also retried. Each retry logs status + wait time to stderr. Closes the 24h-staleness window from a single transient failure.
D-040 (F4)HIGHbothPagination loop now tracks the last next_cursor seen and whether termination was a real end-of-cursor-chain. If the loop exits via pages >= MAX_PAGES while next_cursor was still present, it throws "hit MAX_PAGES=… but next_cursor still present — dataset is truncated". MAX_PAGES raised from 200 → 500 for headroom on heavy backfills.
D-041 (F5)MEDrobustnessSkip the write entirely when nothing material changed. Compares sorted days[], user_id, handle, and bucketed_tz via canonical JSON.stringify; if all four match, logs "no data changes — skipping write" and exits 0 without rewriting generated_at. The workflow's git diff --quiet then correctly no-ops. Avoids ~365 noise commits / Vercel rebuilds per year.
D-042 (F6)MEDgha-securitySHA-pinned the only third-party action: pnpm/action-setup@a7487c7e89a18df4991f7f222e4898a00d66ddda # v3.0.0. Mitigates the moving-major-tag supply-chain hazard for a workflow that has GITHUB_TOKEN + SOCIALDATA_API_KEY in scope. actions/checkout@v4 and actions/setup-node@v4 intentionally left as major-tag pins (first-party).
D-043 (F7)MEDgha-securityCommit step now does git pull --rebase --autostash origin "${GITHUB_REF_NAME}" between git add and git commit. Closes the schedule-after-dispatch race where a queued schedule run would hit non-fast-forward and lose its work. Deviation: brief specified plain git pull --rebase; added --autostash because the rebase runs with staged-but-uncommitted changes in the index, which plain git pull --rebase refuses. Autostash stashes the staged file, rebases, then re-applies — matches the brief's "replay our result on a newer base" semantics exactly.
D-044 (F8)MEDrobustnesssaveData now writes to DATA_PATH + ".tmp" first and then rename()s. Mid-write crash leaves the real file intact rather than truncated/syntactically-invalid.
D-045 (F9)MEDcorrectnesstwitter.ts now tracks shape-valid rows separately from window-filtered rows. If rawDays.length > 0 && shapeOk === 0, throws TwitterFeedOfflineError with reason "all days[] entries failed shape validation" instead of returning an all-zero Day[]. Closes the asymmetric-validator-rot hazard.
D-046 (F10)MEDcorrectnessProducer + consumer now agree on the bucketing tz. (1) refresh-x-days.ts stamps bucketed_tz: TZ into the JSON output and into the DataFile type. (2) Seed x-days.json updated to include "bucketed_tz": "America/Los_Angeles".
