Issue contract

Template fields

.github/ISSUE_TEMPLATE/add-person.yml
  display_name: string, required
  github_login: string, required
  x_login: string, required
  // no optional fields; one-screen self-add form

Maintainer patch

{
  "slug": "github-login-lowercase",
  "displayName": "name",
  "githubLogin": "github-login",
  "xLogin": "x-login"
}

Algorithm

  1. User opens /join.
  2. User clicks the prefilled GitHub issue-template URL.
  3. Maintainer copies fields into apps/web/src/config/users.json.
  4. Vercel rebuild shows GitHub data; refresh workflow creates/fills X data for the new slug.

UI layout

  1. /join: small form asks display/GitHub/X/context, then opens a prefilled GitHub issue.
  2. /: keep existing [ add ] link to /join.

Decisions

IDChoice · Rationale
D-001Use a GitHub issue form over an app write API. File-based roster stays reviewable and needs no new secret.
D-002Migrate X payloads to one keyed JSON file. Current static imports make every new person a code change.
D-003Use maintainer-edited users.json for adds. Direct app commits are more machinery than this roster needs.
D-004Make the issue form self-add only. Three required fields are enough for a roster row.
D-005Ask only roster fields on /join and pass issue-form IDs as query params. Keeps the GitHub issue reviewed while avoiding duplicate typing.

Deviations

X-001Initial plan used POST /api/people; replaced with GitHub issue template; user requested issue-first flow.

Tradeoffs

IDPicked over Rejected · Reason
T-001Issue template over app API. Avoids app-side GitHub write token.
T-002Manual JSON edit over direct commit. Review beats surprise roster writes.
T-003Client-built issue URL over server route. No secret, no storage, no backend.
T-004No explainer after the button over maintainer snippet. Shorter page; docs keep the file-edit details.
